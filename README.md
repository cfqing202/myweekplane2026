# 周月计划小程序 - 设计文档

## 一、项目概述

- **项目名称**：周月计划
- **项目类型**：H5 单页应用
- **部署方案**：GitHub + Vercel（免费外网访问）+ Supabase（数据存储）
- **技术栈**：纯 HTML + CSS + JavaScript + Supabase SDK
- **核心功能**：支持周计划和月计划的添加、删除、全局浏览，支持多用户数据隔离

---

## 二、用户系统需求

### 2.1 登录/退出

- **登录页**：全屏居中白色页面，展示 📋 图标 + 标题 + 手机号输入框 + 登录按钮
- **手机号输入**：
  - 纯数字键盘输入（`inputmode="numeric"`）
  - 最大长度 11 位（`maxlength="11"` + JS 层自动过滤非数字字符）
  - 输入框居中放大显示（`font-size: 20px`，`letter-spacing: 4px`）
  - 支持回车键快捷登录
- **校验**：不足 11 位时显示红色错误提示，不做号码真实性验证
- **登录流程**：
  1. 查询 Supabase `users` 表是否存在该手机号
  2. 不存在则自动创建用户记录
  3. 存在则直接登录
- **登录状态保持**：登录后将手机号存入 `localStorage`（`currentUserPhone`），下次打开自动查询用户ID并进入
- **退出登录**：页面顶部灰色用户栏右侧显示"退出登录"按钮，点击后清除手机号，回到登录页
- **用户脱敏**：用户栏左侧显示 `用户：138****1234`（中间四位隐藏）

### 2.2 数据归属

- 每条计划记录通过 `user_id`（UUID 外键）关联 Supabase `users` 表
- 查询时按 `user_id` 过滤，实现多用户数据隔离
- 退出登录后清除内存数据，重新登录从 Supabase 重新加载

## 三、前端 UI 功能（当前已实现）

### 3.1 整体布局

页面从上到下分为：
1. **用户栏**（登录后）：灰色背景，左侧脱敏用户名，右侧退出按钮
2. **类型 Tab**：`周计划` | `月计划`，居中两个按钮，选中为黑底白字
3. **时间选择区**：2 列网格布局，默认显示 4 个按钮，仅新增时使用
4. **新增按钮**：`+ 新增计划记录`，黑色全宽按钮
5. **计划列表**：按时间分组展示，每组有 📅 标题

### 3.2 计划类型切换

- 顶部两个 Tab：`周计划` / `月计划`
- 切换后刷新时间选择按钮和计划列表
- 列表只显示当前类型的数据

### 3.3 时间选择模块（仅新增时使用）

**周计划默认 4 按钮：**
| 按钮 | 标签示例 | 作用 |
|------|----------|------|
| 上周 | `2026年4月-4周` | 选中后新增记录归入该周 |
| 本周 | `2026年4月-5周` | 默认选中 |
| 下周 | `2026年5月-1周` | 选中后新增记录归入该周 |
| 本月全部 | `本月全部`（灰色边框） | 点击展开为详细周按钮 |

**点击"本月全部"展开为（含 ⬆️ 收起）：**
- `2026年3月-5周`（上月最后一周，默认选中）
- `2026年4月-1周` 至 `2026年4月-5周`（本月所有周，自动计算）
- `⬆️ 收起` 按钮：点击恢复 4 按钮模式

**月计划默认 4 按钮：**
| 按钮 | 标签示例 | 作用 |
|------|----------|------|
| 上月 | `2026年3月` | 选中后新增记录归入该月 |
| 本月 | `2026年4月` | 默认选中 |
| 下月 | `2026年5月` | 选中后新增记录归入该月 |
| 查看全年月份 | `查看全年月份`（灰色边框） | 点击展开为月份列表 |

**点击"查看全年月份"展开为（含 ⬆️ 收起）：**
- `2026年4月` 至 `2026年12月`（当前月到年底，默认选中当前月）
- `⬆️ 收起` 按钮：点击恢复 4 按钮模式

### 3.4 新增记录

- 点击 `+ 新增计划记录` 按钮弹出底部遮罩弹窗
- 弹窗内容：标题"添加计划" + 多行文本输入框（20字限制 + 实时字数统计） + 保存/取消按钮
- **周计划新增**：新增时记录当前选中的周标识（如 `2026年4月-5周`）
- **月计划新增**：新增时记录当前选中的月份标识（如 `2026年4月`），同时显示"是否每月零点自动生成"勾选框
- 保存时调用 Supabase REST API 插入 `plans` 表，获取返回的 UUID 作为记录 ID

### 3.5 每月自动生成（仅月计划）

- 新增月计划时可选勾选 `是否每月零点自动生成`
- 默认为不勾选
- 勾选后，该计划内容每月 0 点自动复制到新月份（去重，避免重复生成）
- 自动生成的记录通过 Supabase API 写入数据库
- 通过 `lastAutoGenerateMonth_<手机号>` 记录最后生成月份（localStorage），防止重复执行

### 3.6 删除记录

- 每条记录右侧显示 ✕ 红色删除按钮
- 点击后调用 Supabase REST API 删除对应记录
- 成功后从内存中移除并刷新列表

### 3.7 计划列表展示

- 按当前 Tab 类型过滤（周计划只显示周数据，月计划只显示月数据）
- 按时间倒序排列（最近的排最前面）
- 按时间分组展示，每组标题为 `📅 2026年4月-5周`（周）或 `📅 2026年4月`（月）
- 组内按添加顺序编号

### 3.8 时间按钮计算规则

- **周数计算**：以周一为一周的第一天，计算日期所在月的第几周
- **上月最后一周**：基于上个月的总天数算出总周数，取最后一周
- **本月各周**：基于本月总天数和第一天星期算出所有周数
- **月份计算**：基于系统当前月份

## 四、数据模型设计

### 4.1 前端本地存储

| key | 格式 | 说明 |
|-----|------|------|
| `currentUserPhone` | `string` | 当前登录用户手机号（11位），用于保持登录状态 |
| `lastAutoGenerateMonth_<phone>` | `string` | 最后自动生成的月份（如 `2026年4月`） |

### 4.2 Supabase 表结构

**表1：`users`（用户表）**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | uuid | 主键，自动生成 |
| phone | text | 手机号码（11位，唯一） |
| created_at | timestamptz | 注册时间 |

**表2：`plans`（计划表）**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | uuid | 主键，自动生成 |
| user_id | uuid | 关联用户ID（外键，ON DELETE CASCADE） |
| type | text | 计划类型：`week` 或 `month` |
| time | text | 时间标识（如 `2026年4月-5周` 或 `2026年4月`） |
| content | text | 计划内容（20字以内） |
| auto_generate | boolean | 是否自动生成模板 |
| template_id | uuid | 关联的模板ID（可空） |
| is_auto_generated | boolean | 是否为自动生成 |
| created_at | timestamptz | 创建时间 |

**索引：**
- `idx_plans_user_id` ON `plans(user_id)`
- `idx_plans_type` ON `plans(type)`

### 4.3 数据映射

前端内存中的计划对象与数据库行之间的映射：

```javascript
// 数据库行 → 内存对象
{
    id: row.id,              // UUID
    type: row.type,          // 'week' | 'month'
    time: row.time,          // '2026年4月-5周'
    content: row.content,    // 计划内容
    autoGenerate: row.auto_generate  // boolean
}

// 内存对象 → 数据库行
{
    user_id: currentUserId,
    type: plan.type,
    time: plan.time,
    content: plan.content,
    auto_generate: plan.autoGenerate,
    is_auto_generated: plan.isAutoGenerated,
    template_id: null
}
```

### 4.4 安全机制

#### RLS 行级安全策略

所有数据通过 Supabase Row Level Security 保护，即使用户自行提取到 anon key 也无法越权访问：

- **users 表**：任何人都可以注册和查询（登录需要）
- **plans 表**：用户只能 SELECT / INSERT / DELETE 自己的计划
- 通过 `current_setting('app.user_phone')` 会话变量在服务端验证当前用户身份

#### 认证流程

```sql
-- 前端调用此函数，告知数据库当前用户身份
SELECT set_user_phone('13800138000');

-- RLS 策略自动验证
-- 等效于：WHERE user_id IN (SELECT id FROM users WHERE phone = '13800138000')
```

#### 安全注意事项

- Supabase anon key 公开是正常设计，安全依赖于 RLS 而非隐藏 key
- 即使有人从源码提取到 anon key + URL，也无法查看或篡改你的计划数据
- 对方需要同时知道你的手机号才能通过 RLS 验证，而手机号不存前端变量中（仅在内存中运行时持有）

## 五、技术方案

### 5.1 前端

- **纯 HTML + CSS + JavaScript**（单页应用，无框架依赖）
- **Supabase JS SDK v2**：通过 CDN 引入（`@supabase/supabase-js`）
- **样式**：PingFang SC / Microsoft YaHei，黑白简约风格

### 5.2 后端（Supabase）

- **数据库**：PostgreSQL，由 Supabase 托管
- **认证**：使用 Supabase `anon key` + Row Level Security
- **API**：Supabase REST API（通过 JS SDK 封装）

### 5.3 部署方案

1. **GitHub 仓库**：托管代码
2. **Vercel**：连接 GitHub 仓库，自动部署静态文件
3. **Supabase**：数据库服务，与前端通过 anon key 直连

---

## 六、部署指南

### 6.1 前提条件

- GitHub 账号
- Vercel 账号（可用 GitHub 账号登录）
- Supabase 项目已创建（数据库表已建好）

### 6.2 部署步骤

**1. 创建 GitHub 仓库**
```bash
git init
git add .
git commit -m "初始化周月计划应用"
git remote add origin https://github.com/你的用户名/weekplane.git
git push -u origin main
```

**2. 连接 Vercel**
- 打开 [vercel.com](https://vercel.com) → Add New Project
- Import GitHub 仓库 `weekplane`
- Framework Preset 选择 **Other**
- Root Directory 保持默认
- 点击 **Deploy**

**3. 部署完成**
- Vercel 会自动分配域名 `https://weekplane.vercel.app`
- 后续每次 push 到 GitHub 都会自动重新部署

### 6.3 本地开发

直接用浏览器打开 `index.html` 即可（需要网络连接加载 Supabase SDK 和连接数据库）。
